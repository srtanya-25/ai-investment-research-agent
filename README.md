# AI Investment Research Agent

A full-stack web app that does first-pass investment research on a company. Enter a
company name and the app reviews its business, growth, and risk, turns that into a
score out of 100, and gives a clear Invest or Pass verdict. Every report is saved and
can be exported as a PDF.

**Live app:** https://ai-investment-research-agent-sandy.vercel.app
**API:** https://invagent-backend.onrender.com
**Demo login:** `demo` / `strongpass1`

## Overview

The goal was a clean, end-to-end research flow rather than a feature-heavy app: log in,
search a company, get three short write-ups (business, growth, and risk), see a single
score and a verdict, keep a history, and export any report. The whole design leans on
being explainable: the score is a transparent rule, so any verdict can be traced back
to the numbers behind it.

### Features

- Sign up and log in (JWT stored in HTTP-only cookies, refreshed automatically)
- Search a company and run a research pass
- Business, growth, and risk analysis generated by the AI layer
- A 0 to 100 investment score and an Invest or Pass verdict
- History of past reports, with the option to delete one
- Export any report to PDF

## How to run it

Requirements: Python 3.12+, Node 20+, and optionally a Gemini API key.

### Backend

```bash
cd backend
python -m venv venv
source venv/bin/activate          # Windows: venv\Scripts\activate
pip install -r requirements.txt

cp .env.example .env               # then fill in the values
python manage.py migrate
python manage.py createsuperuser
python manage.py runserver
```

Runs at http://localhost:8000

### Frontend

```bash
cd frontend
npm install
cp .env.example .env
npm run dev
```

Runs at http://localhost:5173

### Environment variables

Backend (`backend/.env`):

| Key | Purpose |
|-----|---------|
| `SECRET_KEY` | Django secret key |
| `DEBUG` | `True` locally, `False` in production |
| `DATABASE_URL` | `sqlite:///db.sqlite3` for dev, a Postgres URL in production |
| `FRONTEND_URL` | The frontend origin, used for CORS |
| `GEMINI_API_KEY` | Gemini key. Can be left blank (see note below) |
| `GEMINI_MODEL` | Defaults to `gemini-2.5-flash` |

Frontend (`frontend/.env`):

| Key | Purpose |
|-----|---------|
| `VITE_API_URL` | Base URL of the backend |

If no Gemini key is set, the app still runs. It falls back to a deterministic offline
analysis, which keeps local development and tests working without a key. With a key set,
the analysis comes from the live model.

## How it works

The backend is Django REST Framework, the frontend is React (Vite). They communicate
over JSON, and the login tokens live in HTTP-only cookies so the frontend JavaScript
never handles them directly.

The layers are kept separate on purpose:

- views stay thin: read the request, call a service, return the result
- services hold the logic and walk through a research run
- the scoring rule is its own small module, easy to test and explain
- everything to do with the model lives in one `ai/` package, and nothing else imports it

A research run flows like this:

```
company name -> ai layer (LangChain + Gemini) -> score -> verdict -> save -> show
```

The AI layer asks Gemini for a structured JSON reply (sector, three analyses, and three
sub-scores), and a parser pulls it apart. If the call fails for any reason, the offline
fallback takes over so the request never breaks.

More detail is in [docs/ARCHITECTURE.md](docs/ARCHITECTURE.md), the endpoints are listed
in [docs/API.md](docs/API.md), and the problems faced during the build are written up in
[docs/KNOWN_ISSUES.md](docs/KNOWN_ISSUES.md).

### How the score is calculated

```
overall = business * 0.40 + growth * 0.35 + (100 - risk) * 0.25
verdict = Invest if overall >= 60, else Pass
```

Risk is inverted because a riskier company should score lower. Every value is clamped to
the 0 to 100 range. Keeping this as a plain rule rather than a learned model means each
verdict can be explained directly.

## Key decisions & trade-offs

- **One model call, not an agent loop.** A single structured prompt returns all three
  analyses and the sub-scores. It is cheaper, faster, and far easier to reason about than
  a multi-step agent.
- **A rule-based score, not a learned one.** The weighting is transparent and easy to
  defend, which matters more here than squeezing out accuracy.
- **A fallback instead of a hard failure.** A missing or rate-limited key drops to offline
  output rather than taking the app down.
- **API over web scraping.** The research comes from the Gemini API rather than scraping
  pages. An API gives clean, supported data; scraping would be fragile and harder to
  maintain. The trade-off is that the analysis reflects model knowledge rather than
  real-time figures.
- **Left out on purpose:** real-time market data, vector search, portfolio features, and
  any complex financial modelling. They would add a lot of surface area for little gain at
  this scope.

## Example runs

With a live key:

- Infosys -> sector: IT Services, verdict Invest, score 69
- Spotify -> sector: Digital Audio Streaming, verdict Invest

Without a key (offline fallback, derived deterministically from the name):

- Acme Corp -> Invest, score 64
- Globex -> Pass, score 52

## What would be improved with more time

- Bring in real financial data (revenue, margins) to ground the scores instead of relying
  only on the model's read
- Let the user adjust the weighting and watch the verdict change
- Cache recent reports per company to avoid repeat model calls for the same name
- Alerts when a re-run changes a saved company's verdict

## Tech stack

React (Vite), Django REST Framework, PostgreSQL, LangChain with Gemini, JWT cookies,
Docker, GitHub Actions, deployed on Vercel (frontend) and Render (backend and database).
